# DB 전면 개편 — 채팅/설정 Supabase 이전 결정사항 & SQL 마이그레이션

## 배경

현재 로컬 파일 기반인 두 저장소를 Supabase로 옮기는 작업.

- `electron/chatStore.js` → `userData/chats.json`. 세션 = `contentId` 하나당 메시지 배열 + provider.
- `electron/settingsStore.js` → `userData/settings.json`. API 키, 모델, 기본 옵션, 카테고리, activeFolder 등.

목적: 기기(설치본)가 바뀌어도 같은 계정으로 로그인하면 채팅 기록/설정이 그대로 보이게.

## 로컬에 남겨야 하는 것 (이전 대상 아님)

- `supabase.url` / `supabase.anonKey`: Supabase에 접속하기 위한 정보 자체라 Supabase 안에 저장하면 순환 참조(닭이 먼저냐 달걀이 먼저냐) 발생. 지금처럼 `settings.json` 로컬 유지.
- `authStorage`(로그인 세션/리프레시 토큰): 세션은 기기별로 독립적이어야 함 — 다른 기기의 리프레시 토큰을 공유하면 로그아웃/보안 모델이 꼬임. docs/260708_183959 supabase 인증 결정사항.md에서 이미 로컬 평문 저장으로 정함, 이건 그대로 유지.

## 동시성(concurrency) 문제 — 검토한 옵션

같은 계정으로 여러 기기에서 동시에 앱을 켜두면 발생 가능한 문제. 옵션 4가지 검토:

1. **Last-write-wins, 통째로 덮어쓰기** — 락도 병합도 없음. 가장 단순. 두 기기가 동시에 수정하면 나중에 저장한 쪽이 이김, 먼저 것 조용히 유실.
2. **낙관적 동시성 제어(버전/updated_at 비교 후 충돌시 거부)** — 유실은 막지만, 충돌 시 사용자에게 뭘 보여줄지(재시도/병합 UI) 추가 구현 필요.
3. **Supabase Realtime 구독 기반 실시간 동기화** — 진짜 멀티 디바이스 동시 편집 지원. 구현 비용 가장 큼(구독 해제/재연결, 로컬 상태와 원격 상태 병합 로직 등).
4. **기기별 분리(공유 안 함)** — 애초에 동시성 문제 자체가 없음. 근데 이번 요구사항(설정/채팅 공유)과 반대라 제외.

**결정 (사용자 지정)**: 1번, last-write-wins. 동시 사용은 가정하지 않음 — 잠금/병합/충돌 감지 일체 구현 안 함. 실질적 의미:

- **설정(`user_settings`)**: 앱은 항상 row 전체를 읽어와서 필드 하나 바꿀 때마다 그 필드 포함 전체를 다시 upsert(현재 로컬 `settingsStore.updateSettings`가 하는 방식과 동일한 패턴). 기기 A, B가 동시에 열려 있다가 각자 다른 필드를 고치고 저장하면, 나중에 저장한 쪽이 앞선 변경사항까지 통째로 덮어씀 — 필드 단위 병합 없음.
- **채팅(`chat_sessions`)**: 메시지는 매번 append라 잦은 충돌 가능성은 낮지만, 같은 세션에 두 기기가 동시에 메시지를 보내면 하나가 유실될 수 있음(SELECT 후 append해서 UPDATE하는 read-modify-write라 원자적이지 않음).

이 정도 리스크는 감수하는 걸로 확정.

## SQL 마이그레이션 (기존 스키마 대비)

전제: docs/260708_183959 supabase 인증 결정사항.md에서 만든 `contents.user_id` + auth 인프라가 이미 적용되어 있음.

```sql
-- ── 채팅 세션 ──────────────────────────────────────────
-- 로컬 chats.json의 { sessions: { [contentId]: { messages, provider, updatedAt } } } 구조를
-- content_id 당 한 행으로 옮김. contents 삭제 시 같이 삭제(discard 시 동작과 동일).
create table chat_sessions (
  content_id bigint primary key references contents(id) on delete cascade,
  user_id uuid not null references auth.users(id) on delete cascade default auth.uid(),
  messages jsonb not null default '[]'::jsonb,
  provider text,
  updated_at timestamptz not null default now()
);

alter table chat_sessions enable row level security;

create policy "select own chat_sessions"
  on chat_sessions for select
  using (auth.uid() = user_id);

create policy "insert own chat_sessions"
  on chat_sessions for insert
  with check (auth.uid() = user_id);

create policy "update own chat_sessions"
  on chat_sessions for update
  using (auth.uid() = user_id)
  with check (auth.uid() = user_id);

create policy "delete own chat_sessions"
  on chat_sessions for delete
  using (auth.uid() = user_id);


-- ── 사용자 설정 ────────────────────────────────────────
-- 로컬 settings.json 중 supabase{url,anonKey}, authStorage 를 제외한 나머지.
-- 사용자당 한 행(1:1) — upsert(on conflict user_id do update)로 항상 전체 갱신.
create table user_settings (
  user_id uuid primary key references auth.users(id) on delete cascade default auth.uid(),
  api_keys jsonb not null default '{}'::jsonb,
  models jsonb not null default '{}'::jsonb,
  default_provider text not null default 'claude',
  default_options jsonb not null default '{}'::jsonb,
  categories jsonb not null default '[]'::jsonb,
  active_folder text,
  updated_at timestamptz not null default now()
);

alter table user_settings enable row level security;

create policy "select own user_settings"
  on user_settings for select
  using (auth.uid() = user_id);

create policy "insert own user_settings"
  on user_settings for insert
  with check (auth.uid() = user_id);

create policy "update own user_settings"
  on user_settings for update
  using (auth.uid() = user_id)
  with check (auth.uid() = user_id);
```

### 기존 로컬 데이터 이전

`chats.json` / `settings.json`은 자동 마이그레이션 스크립트 없음(로컬 파일 → 원격 DB 매핑이라 SQL만으로 불가능). 앱 소유자가 최초 로그인 시, 1회성으로 기존 파일을 읽어 위 테이블에 insert하는 코드가 필요함 — 이건 SQL이 아니라 구현 단계(마이그레이션 스크립트/앱 코드)에서 처리.

## 확인 필요

- API 키(`api_keys`)를 Supabase에 평문 jsonb로 저장하는 것 — 지금 로컬 `settings.json`도 평문이라 저장 수준 자체는 동일하지만, "네트워크로 나간다"는 차이는 있음(Supabase는 TLS로 전송되고 `anon` 키로 인증된 사용자 본인만 RLS로 읽을 수 있음). 이대로 진행해도 되는지.
- `chat_sessions.messages`를 통째 jsonb 배열로 두는 건 기존 로컬 구조 그대로 옮긴 것 — 메시지 개별 행으로 정규화할지, 지금처럼 배열 통째로 둘지(더 단순, read-modify-write라 위 동시성 리스크 있음) 확인.
- 이 마이그레이션까지 적용하고 실제 코드 구현(`chatStore.js`/`settingsStore.js`를 `db.js`처럼 Supabase 클라이언트 기반으로 교체) 바로 진행할지, 아니면 SQL 검토만 먼저 하고 넘어갈지.
