# Supabase 전환

## 변경 내용

- `electron/db.js`: sql.js(로컬 파일 DB) → `@supabase/supabase-js` 클라이언트로 전면 교체. IPC 인터페이스(`insertContent`, `updateContent`, `listByStatus`, `approve`, `discard`, `resetStuckJobs`)는 그대로 유지해 `electron/main.js` 쪽 변경은 최소화됨.
- `electron/settingsStore.js`: 설정에 `supabase: { url, anonKey }` 필드 추가. `settings.json`(userData 폴더)에 평문 저장됨 — 기존 API key 저장 방식과 동일한 수준.
- `src/pages/Settings.tsx`: Database(Supabase) 섹션 추가, Project URL / anon key 입력.
- `src/hooks/useSupabaseConfig.ts`: 신규 훅, `window.api.getSettings/syncSettings` IPC로 supabase 설정 읽고 씀.
- `src/types/global.d.ts`: `PipelineSettings.supabase` 타입 추가.
- `package.json`: `sql.js`, `@types/sql.js` 제거, `@supabase/supabase-js` 추가.
- `electron/main.js`: `db.insertContent` 실패(네트워크 미설정 등) 시 앱이 죽지 않도록 `processLink` 시작부에 try/catch 추가. `resetStuckJobs`도 startup에서 실패해도 무시하고 진행하도록 catch 추가.

이미지(`data.thumbnail`)는 요청대로 URL 문자열 그대로 저장 — DB 스키마/데이터 변경 없음. 캐싱은 별도 이슈(아래) 참고.

## 사용자가 직접 해야 하는 작업 (앱이 대신 못함)

### 1. Supabase 프로젝트에 테이블 생성

Supabase 대시보드 → SQL Editor 에서 실행:

```sql
create table contents (
  id bigint generated always as identity primary key,
  url text not null,
  tag text not null,
  status text not null default 'pending',
  data jsonb not null default '{}'::jsonb,
  created_at timestamptz not null default now()
);
```

### 2. RLS(Row Level Security) 처리

Supabase는 테이블 생성 시 기본적으로 RLS를 켜두는 옵션이 있음. anon key로 앱이 직접 CRUD 하는 구조라, 아래 둘 중 하나 필요:

- (간단) RLS 끄기: `alter table contents disable row level security;`
- (권장, 단일 사용자 개인 앱이라도 최소한의 안전장치) 전체 허용 정책 추가:
  ```sql
  alter table contents enable row level security;
  create policy "allow all" on contents for all using (true) with check (true);
  ```

앱은 단일 사용자용 로컬 데스크톱 앱이고 anon key가 클라이언트에 박혀있는 구조라, RLS로 막아봤자 실질적 보안 이득은 없음(키 자체가 노출됨). 다만 향후 anon key 유출 시 제3자가 테이블을 읽고 쓸 수 있다는 점은 인지 필요.

### 3. 앱 설정 화면에 입력

Settings 페이지 → Database(Supabase) 섹션에서:
- Project URL: `https://xxxxx.supabase.co` (Supabase 대시보드 → Project Settings → API)
- anon key: 같은 페이지의 `anon public` key

미입력 상태로 앱 쓰면 DB 관련 동작(pending/approve/기사 저장 등) 전부 에러 처리됨(`Supabase is not configured...`) — 크래시는 안 나지만 기능 동작 안 함.

## 이미지 캐싱

`data.thumbnail`은 원본 기사 사이트의 이미지 URL 그대로 저장(요청사항 유지). 대신 렌더링 경로에 로컬 디스크 캐시 계층 추가:

- `electron/imageCache.js`: `appimg://` 커스텀 프로토콜 핸들러. 최초 요청 시 원본 URL을 fetch해 `userData/image-cache/`에 저장(내용 + content-type), 이후 요청은 디스크에서 바로 서빙.
- `src/utils/imageCache.ts`: `cachedImageSrc(url)` → `appimg://${encodeURIComponent(url)}` 변환. `Pending.tsx`, `Archive.tsx`의 `<img src>` 전부 이걸 통해서 렌더링.
- DB에는 캐시 경로가 아니라 원본 URL만 저장됨 — 캐시는 순수 로컬 렌더링 최적화 계층이라 Supabase 스키마와 무관.

캐시 무효화(예: 오래된 이미지 정리)는 별도 구현 안 함 — 필요하면 추가 요청.
