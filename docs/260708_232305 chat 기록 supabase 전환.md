# Chat 기록 Supabase 전환 — 마이그레이션 SQL & 구현 요약

> 스키마/동시성 정책은 이미 `article-sum/docs/260708_185419 db 전면 개편 결정사항.md`에서 결정됨. 이 문서는 그 결정 중 채팅 부분만 실행 가능한 형태로 다시 정리 + 실제 코드 반영 내용 기록.

## 배경

`article-sum/electron/chatStore.js`가 로컬 파일(`userData/chats.json`)에 채팅 기록을 저장하던 것을 Supabase `chat_sessions` 테이블로 전환. 기존 로컬 파일 기록은 보존하지 않음(요구사항) — 별도 백필 스크립트 없음.

## SQL 마이그레이션 (Supabase SQL Editor에서 실행)

이미 `260708_185419` 문서의 "2. 채팅/설정 이전" 섹션에서 실행했다면 스킵. 신규로 적용하는 경우:

```sql
create table chat_sessions (
  content_id bigint primary key references contents(id) on delete cascade,
  user_id uuid not null references auth.users(id) on delete cascade default auth.uid(),
  messages jsonb not null default '[]'::jsonb,
  provider text,
  updated_at timestamptz not null default now()
);

alter table chat_sessions enable row level security;

create policy "select own chat_sessions"
  on chat_sessions for select
  using (auth.uid() = user_id);

create policy "insert own chat_sessions"
  on chat_sessions for insert
  with check (auth.uid() = user_id);

create policy "update own chat_sessions"
  on chat_sessions for update
  using (auth.uid() = user_id)
  with check (auth.uid() = user_id);

create policy "delete own chat_sessions"
  on chat_sessions for delete
  using (auth.uid() = user_id);
```

`content_id`가 `contents(id)`를 참조하므로 `contents` row가 삭제(Discard)되면 연결된 `chat_sessions` row도 `on delete cascade`로 자동 삭제됨 — 앱 코드에서 별도로 `chatStore.deleteSession()`을 호출하는 것과 별개로 안전망 역할.

기존 로컬 `chats.json` → 위 테이블 백필: **없음.** 기존 기록 보존 불필요하다는 지시에 따라 마이그레이션 스크립트 작성 안 함. 로컬 `chats.json` 파일 자체는 그대로 두거나(미사용 상태) 사용자가 원하면 수동 삭제.

## 동시성 정책

`260708_185419` 문서와 동일: last-write-wins, read-modify-write. 메시지 append 시 `messages` jsonb 배열 전체를 읽고 → JS에서 push → 전체 upsert. 같은 세션에 두 기기가 동시에 메시지를 보내면 하나가 유실될 수 있음 — 감수하기로 확정된 리스크.

## 앱 코드 변경

- `article-sum/electron/db.js`: `getClient`를 `module.exports`에 추가 — `chatStore.js`가 동일 Supabase 클라이언트를 재사용하기 위함.
- `article-sum/electron/chatStore.js`: `fs`/`userData/chats.json` 기반 구현 전면 삭제, `chat_sessions` 테이블 기반 구현으로 교체. 함수 시그니처(`getSession`, `listSessions`, `appendMessage`, `setProvider`, `deleteSession`)는 동일하게 유지하되 전부 `async`로 전환.
- `article-sum/electron/main.js`: `chatStore`가 이제 비동기이므로 `contents:discard`와 `chat:send` 핸들러 내 호출부에 `await` 추가(`chat:get`/`chat:list`/`chat:delete`는 promise를 그대로 반환하던 구조라 수정 불필요).

## 확인 필요

- Supabase 프로젝트에 `chat_sessions` 테이블이 아직 없다면 위 SQL을 먼저 실행해야 앱이 정상 동작함(없는 상태로 채팅 시도 시 `chat:send`가 에러로 실패).
