# Supabase 인증(로그인) 도입 — 결정사항 & SQL 마이그레이션

## 배경

현재 상태 (docs/260708_155204 supabase 전환.md):
- `contents` 테이블 하나, RLS 꺼져있거나 "allow all" 정책 — 사실상 인증 없음.
- Electron **main process**(`electron/db.js`)에서 `@supabase/supabase-js` client 생성, anon key는 `settings.json`에 평문 저장.
- 채팅 세션(`electron/chatStore.js`)은 로컬 `chats.json` 파일 — Supabase와 무관, 이번 인증 작업 범위 밖.

## 확정된 결정사항 (사용자 답변)

1. **인증 목적**: 멀티유저 — 로그인한 사용자별로 `contents` 데이터 분리.
2. **로그인 방식**: Email + Password. (OAuth 검토했으나 Electron에서 브라우저 왕복/딥링크 콜백 필요해 복잡도 높아 기각.)
3. **가입 정책**: 앱 내 가입 허용 (`signUp` 폼 제공).
4. **이메일 확인(confirmation)**: 끔 — 가입 즉시 로그인 가능. 이메일 소유권 검증은 안 함(개인용 앱이라 문제없음). 이 덕분에 메일 링크 클릭 → 앱 복귀용 딥링크/loopback 콜백이 아예 필요 없어짐.
5. **세션(리프레시 토큰) 저장**: `settings.json`에 평문 저장 — 기존 anon key와 동일 방식, 커스텀 storage adapter로 supabase-js에 연결.

### 기존 `contents` 데이터 마이그레이션

기존 행엔 `user_id`가 없음. 새 정책 적용 시 소유자 없는 행은 아무한테도 안 보임.

`user_id`를 nullable로 추가 → 앱 소유자가 로그인 후 1회 수동으로 자기 계정에 귀속(아래 마이그레이션 SQL 하단 참고) → 이후 `NOT NULL` 제약 추가.

### 세션 만료 시 UX

Supabase는 refresh token으로 자동 갱신하므로 평소엔 신경 안 써도 됨. 리프레시 토큰 자체가 만료/무효화된 경우만 재로그인 화면으로 보내면 됨 — `onAuthStateChange`의 `SIGNED_OUT` 이벤트로 처리.
