# Supabase 인증(로그인) 도입 — 결정사항 & SQL 마이그레이션

## 배경

현재 상태 (docs/260708_155204 supabase 전환.md):
- `contents` 테이블 하나, RLS 꺼져있거나 "allow all" 정책 — 사실상 인증 없음.
- Electron **main process**(`electron/db.js`)에서 `@supabase/supabase-js` client 생성, anon key는 `settings.json`에 평문 저장.
- 채팅 세션(`electron/chatStore.js`)은 로컬 `chats.json` 파일 — Supabase와 무관, 이번 인증 작업 범위 밖.

## 확정된 결정사항 (사용자 답변)

1. **인증 목적**: 멀티유저 — 로그인한 사용자별로 `contents` 데이터 분리.
2. **로그인 방식**: OAuth (Google).
3. **가입 정책**: 앱 내 가입 허용 (OAuth라 Google 계정 있으면 첫 로그인 = 자동 가입).

## 추가로 필요한 결정 (엔지니어링 사항 — 아래는 제안 기본값)

### 4. Electron에서 OAuth 콜백을 어떻게 받을지

Google은 Electron `BrowserWindow`(embedded webview)로 로그인하는 걸 정책상 막는 경우 많음. 표준 데스크톱 패턴:

- 시스템 기본 브라우저로 Supabase OAuth URL 오픈 (`supabase.auth.signInWithOAuth`가 주는 url을 `shell.openExternal`).
- 커스텀 프로토콜(`clipbrief://auth-callback`)을 리다이렉트 URL로 등록 → OS가 로그인 완료 후 이 URL로 앱을 다시 실행/포커스.
- `main.js`에서 `app.setAsDefaultProtocolClient('clipbrief')` + `second-instance`/`open-url` 이벤트로 콜백 URL(코드/토큰 포함) 캐치 → `supabase.auth.exchangeCodeForSession()`.

**제안**: 위 방식 채택, 스킴 이름 `clipbrief`. 단, 지금 `package.json`에 electron-builder 설정 자체가 없어서(패키징 안 된 상태, `electron .`로만 실행) 프로토콜 등록이 정식으로 동작하려면 electron-builder 도입 + 빌드 필요 — 이건 이번 인증 작업과 별개로 먼저 처리할지 확인 필요.

### 5. 세션(리프레시 토큰) 저장 위치

Main process엔 `localStorage`가 없어서 supabase-js에 커스텀 storage adapter 필요.

- (A) 기존 anon key처럼 `settings.json`에 평문 저장 — 구현 간단, 기존 방식과 일관.
- (B) Electron `safeStorage`(OS 자격 증명 저장소, DPAPI on Windows)로 암호화 저장 — 더 안전, 코드 조금 더 필요.

**제안**: (B). 로그인 토큰은 anon key보다 민감도 높음(사용자 개인 데이터 접근권) — `safeStorage` 있는 만큼 쓰는 게 맞음.

### 6. 기존 `contents` 데이터 마이그레이션

기존 행엔 `user_id`가 없음. 새 정책 적용 시 소유자 없는 행은 아무한테도 안 보임.

**제안**: `user_id`를 nullable로 추가 → 앱 소유자가 로그인 후 1회 수동으로 자기 계정에 귀속(아래 마이그레이션 SQL 하단 참고) → 이후 `NOT NULL` 제약 추가.

### 7. 세션 만료 시 UX

Supabase는 refresh token으로 자동 갱신하므로 평소엔 신경 안 써도 됨. 리프레시 토큰 자체가 만료/무효화된 경우만 재로그인 화면으로 보내면 됨 — 별도 결정 불필요, `onAuthStateChange`의 `SIGNED_OUT` 이벤트로 처리.

## SQL 마이그레이션 (기존 스키마 대비)

기존 스키마 (docs/260708_155204):

```sql
create table contents (
  id bigint generated always as identity primary key,
  url text not null,
  tag text not null,
  status text not null default 'pending',
  data jsonb not null default '{}'::jsonb,
  created_at timestamptz not null default now()
);
```

### 마이그레이션

```sql
-- 1. user_id 컬럼 추가 (일단 nullable — 기존 데이터 마이그레이션 위해)
alter table contents
  add column user_id uuid references auth.users(id) on delete cascade
  default auth.uid();

-- 2. 기존 "allow all" 정책 제거 (있다면)
drop policy if exists "allow all" on contents;

-- 3. RLS 켜기
alter table contents enable row level security;

-- 4. 사용자별 정책 (자기 데이터만 CRUD)
create policy "select own contents"
  on contents for select
  using (auth.uid() = user_id);

create policy "insert own contents"
  on contents for insert
  with check (auth.uid() = user_id);

create policy "update own contents"
  on contents for update
  using (auth.uid() = user_id)
  with check (auth.uid() = user_id);

create policy "delete own contents"
  on contents for delete
  using (auth.uid() = user_id);
```

### 기존 데이터 귀속 (앱 소유자 1회 실행)

로그인 후 Supabase 대시보드 → Authentication → Users에서 본인 계정 UUID 확인 후:

```sql
update contents
set user_id = '<본인 auth.users.id>'
where user_id is null;
```

### 백필 끝난 뒤 (선택, 데이터 무결성 강제하고 싶으면)

```sql
alter table contents alter column user_id set not null;
```

## Supabase 대시보드에서 직접 해야 하는 작업

1. **Authentication → Providers → Google** 활성화. Google Cloud Console에서 OAuth Client ID/Secret 발급 후 등록. Authorized redirect URI는 Supabase가 안내하는 `https://<project-ref>.supabase.co/auth/v1/callback`.
2. **Authentication → URL Configuration → Redirect URLs**에 `clipbrief://auth-callback` 추가 (스킴 이름 확정 후).
3. 위 SQL 마이그레이션 SQL Editor에서 실행.
