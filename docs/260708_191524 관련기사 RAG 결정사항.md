# 관련 기사(RAG) 기능 — 결정 필요 사항

> 요구사항: Archive의 스크랩된(승인된) 기사를 펼치면 "관련 기사" 섹션 표시. `processLink()`에서 임베딩 생성 → Supabase pgvector에 저장 → 코사인 유사도 topK=5로 검색 → 가로 스크롤 row에 카드 5개로 표시.

아직 구현 안 함. 아래는 구현 전 확정해야 하는 결정 사항 목록.

## 현재 코드 기준 사실관계

- `electron/main.js`의 `processLink(url)`이 크롤링(`crawlWithRetry`) → 요약(`summarizeArticle`, `electron/llm.js`) 순서로 실행. 이 사이 어딘가에 임베딩 단계 삽입 필요.
- 요약/채팅에 쓰는 프로바이더는 4개: `claude`, `gemini`, `openai`, `nvidia`(`src/types/index.ts`). **Claude는 임베딩 API 자체가 없음.**
- API 키는 `settings.apiKeys[provider]`에 프로바이더별 1개씩 저장(`electron/settingsStore.js`). 임베딩 전용 키/모델 필드는 없음.
- `contents` 테이블(Supabase)은 `id, url, tag, status, data(jsonb), user_id, created_at` 구성 (`electron/db.js`, docs의 DB 개편 문서 참고). pgvector 확장/컬럼 아직 없음.
- Archive.tsx는 `status='approved'`인 것만 조회(`listApproved`)해서 폴더별로 그룹핑, 펼치면(`expanded`) 상세 뷰.

## 결정 사항

### 1. 임베딩에 쓸 프로바이더/모델
Claude는 제외 대상(임베딩 API 없음). 후보:
- **OpenAI** `text-embedding-3-small` (1536차원, 저렴)
- **Gemini** `text-embedding-004` (768차원)
- **NVIDIA NIM** 임베딩 모델(예: `nv-embed-v1`, 차원 다름)

문제: 사용자의 `defaultProvider`가 Claude면 임베딩 못 함. 옵션:
- (a) 임베딩 전용 프로바이더를 고정(예: 항상 OpenAI 키 사용) — 그 키 없으면 임베딩 skip.
- (b) `defaultProvider` 따라가되 Claude일 때만 폴백 프로바이더 사용.
- (c) Settings에 "임베딩 프로바이더" 별도 선택 항목 신설.

→ **어느 걸로 할지 확정 필요.** 벡터 차원이 여기서 결정되므로 DB 컬럼 스키마(4번)에도 영향.

### 2. 임베딩 대상 텍스트
- **원문(`data.original`) 전체**: 정확도는 높지만 임베딩 모델 토큰 한도(예: OpenAI 8191 토큰) 넘는 기사 있음 → truncate 필요.
- **제목 + 요약(`data.summaries`)**: 짧고 안전하지만, 요약은 사용자가 고른 언어/이모지 옵션에 따라 내용이 달라짐(`computeOptionKey`로 옵션별 여러 버전 존재) — 어떤 버전을 쓸지 애매하고, 언어가 기사마다 다르면 유사도 비교가 흔들릴 수 있음.
- **제목만**: 가장 단순하지만 정보 손실 큼.

→ 확정 필요. (원문 앞 N자 truncate 방식이 무난해 보임 — 제안이지 결정 아님)

### 3. 임베딩 시점 & 실패 처리
- 크롤링 성공 직후(원문 확보 시점) vs 요약 완료 후(카테고리/요약까지 다 있을 때). 2번 답이 "원문"이면 크롤링 직후도 가능, "요약"이면 요약 이후여야 함.
- 임베딩 API 호출 실패 시 `processLink()` 전체를 실패 처리할지, 아니면 임베딩만 조용히 스킵(기존 스크랩 흐름은 그대로 성공)하고 관련기사 검색에서만 빠지게 할지.

→ 실패해도 스크랩 자체는 막지 않는(fail-soft) 쪽 제안 — 확정 필요.

### 4. DB 스키마
```sql
-- 예시, 차원수는 1번 결정에 종속
alter table contents add column embedding vector(1536);
```
- `create extension if not exists vector;` 필요(Supabase 프로젝트에서 보통 대시보드 Database → Extensions에서 켜야 함).
- 인덱스(`ivfflat`/`hnsw`, cosine ops): 기사 수 적을 땐 없어도 되지만 나중에 필요. 지금 만들지, 나중에 데이터 쌓이면 만들지.
- 검색은 클라이언트에서 전체 벡터 끌어와 JS로 코사인 계산 vs **Postgres RPC 함수**(`<=>` 연산자로 DB 안에서 topK 계산). 후자가 정석 — RPC 함수 새로 작성 필요.

### 5. 검색 범위
- `status='approved'`(Archive에 보이는 것)만 대상인지, `pending`도 포함인지 — "저장된 기사"라는 표현상 approved만이 맞아 보이나 확인 필요.
- 본인 소유(`user_id`) 기사만 검색되게 할 것(RLS). RPC 함수를 `SECURITY DEFINER`로 만들면 RLS를 우회하므로 함수 안에서 `user_id = auth.uid()` 조건을 직접 넣어야 함 — 아니면 `SECURITY INVOKER`로 두고 RLS에 맡기는 방식.
- 펼친 기사 자기 자신은 결과에서 제외.
- 유사도 임계값(threshold) 둘지: topK=5 고정이면 기사가 몇 개 없거나 다 안 비슷해도 5개(또는 그 이하) 억지로 나옴. 최소 유사도 기준 둘지, 아니면 "가장 가까운 것부터 최대 5개" 그대로 둘지.

### 6. IPC/UI
- 새 IPC 채널 필요(예: `contents:related`) — `preload.js`, `electron/main.js` 핸들러, `electron/db.js` RPC 호출, `src/types/global.d.ts`의 `ElectronApi`, `Archive.tsx` 추가.
- 관련 기사 카드 클릭 시 동작: 그 기사로 스크롤/펼치기 이동할지, 별도 모달(기존 `fullTextRecord` 모달 재사용?) 띄울지.
- 관련 기사 0개일 때 UI 문구 표시 여부(빈 섹션 숨김 vs "관련 기사 없음" 표시).
- 관련 기사 목록 로딩 시점: 펼칠 때마다 매번 API 호출(row 접었다 펼 때마다 재조회) vs 최초 펼칠 때 1번만 fetch해서 캐시.

### 7. 기존 데이터 백필
- 이 기능 배포 전에 이미 승인된 기사들은 `embedding`이 없음. 소급 계산(백필 스크립트/1회성 배치) 할지, 아니면 이 기능 이후 새로 스크랩되는 기사부터만 관련기사 대상으로 할지.

### 8. 비용
- 스크랩 1건마다 임베딩 API 호출 1회 추가(요약 API와 별도 과금). 사용자에게 별도 안내 필요 없다고 보는지 확인.

## 확정된 결정 (사용자 지정)

1. **임베딩 프로바이더**: OpenAI 고정 (`text-embedding-3-small`, 1536차원). `settings.apiKeys.openai` 없으면 임베딩 스킵(스크랩 자체는 진행).
2. **임베딩 입력 텍스트**: 원문(`data.original`) 앞부분 truncate. 6000자로 자름(한글 기준 토큰 한도 8191 안전 마진).
3. **실패 처리**: fail-soft — 임베딩 실패해도 스크랩/요약은 그대로 성공 처리. 대신 `data.embeddingError`에 에러 메시지 기록해서 항목에 표시(관련기사 검색 대상에서만 빠짐).
4. **검색 범위**: `status='approved'`만, 자기 자신 제외, 최소 유사도 threshold 적용(미달이면 5개 안 채우고 그만큼만 표시).

## 남은 항목 중 기본값으로 진행한 것 (구현 시 조정 가능한 상수)

- 유사도 threshold: 코사인 유사도 **0.75** 이상만 후보 (`match_contents` SQL 함수 파라미터, 값은 조정 가능).
- 인덱스: 초기엔 안 만듦(데이터 적음). 나중에 기사 수 늘어나면 `ivfflat`/`hnsw` 인덱스 추가 검토.
- RPC 함수는 `SECURITY INVOKER`(기본값) — RLS(`select own contents` 정책)가 그대로 적용되어 본인 소유 기사만 검색됨.
- 기존(배포 전) 저장된 기사 백필 안 함 — 이 기능 이후 새로 스크랩되는 기사부터 임베딩 생성. 과거 기사는 관련기사 검색에서 서로 안 잡힘(embedding NULL).
- 관련 기사 fetch 시점: Archive에서 펼칠 때 1회 fetch, 클라이언트 state에 캐시(다시 접었다 펴도 재요청 안 함).
- 관련 기사 카드 클릭 시: 기존 "View full text" 모달 재사용.

## SQL 마이그레이션

```sql
-- pgvector 확장 활성화
create extension if not exists vector;

-- contents 테이블에 임베딩 컬럼 추가 (OpenAI text-embedding-3-small = 1536차원)
alter table contents add column embedding vector(1536);

-- 코사인 유사도 기반 관련기사 검색 함수
-- source_id 기사의 임베딩과 가까운 approved 기사 최대 match_count개 반환
create or replace function match_contents(
  source_id bigint,
  match_count int default 5,
  match_threshold float default 0.75
)
returns table (
  id bigint,
  url text,
  tag text,
  status text,
  data jsonb,
  created_at timestamptz,
  similarity float
)
language sql
stable
as $$
  select c.id, c.url, c.tag, c.status, c.data, c.created_at,
         1 - (c.embedding <=> s.embedding) as similarity
  from contents c, (select embedding from contents where id = source_id) s
  where c.status = 'approved'
    and c.id <> source_id
    and c.embedding is not null
    and s.embedding is not null
    and 1 - (c.embedding <=> s.embedding) >= match_threshold
  order by c.embedding <=> s.embedding
  limit match_count;
$$;
```

RLS는 기존 `contents` 테이블 정책(`select own contents` 등)을 그대로 씀 — 함수가 `SECURITY INVOKER`(기본값)라 호출한 사용자 권한으로 실행되어 자기 소유 기사만 조회됨. 새 정책 추가 불필요.

## 구현 범위

- `electron/llm.js`: `embedText(text, key, signal)` 추가 (OpenAI `/embeddings` 호출).
- `electron/db.js`: `updateContent`에 `embedding` 파라미터 추가, `getRelated(id)` 추가(`match_contents` RPC 호출).
- `electron/main.js`: `processLink()` 크롤링 직후 임베딩 단계 삽입(fail-soft), `contents:related` IPC 핸들러 추가.
- `electron/preload.js`: `getRelated` 노출.
- `src/types/global.d.ts`: `ContentRecord.data.embeddingError`, `ElectronApi.getRelated` 추가.
- `src/pages/Archive.tsx`: 펼친 뷰에 "관련 기사" 가로 스크롤 row 추가, 임베딩 실패 배지 표시.

## 사용자 액션 필요

위 SQL을 Supabase SQL Editor에서 실행해야 기능이 동작함(앱 코드가 자동으로 마이그레이션하지 않음).
